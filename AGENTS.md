# AI News Codex 工作规则

## 项目定位

本项目用于维护一个小红书 AI 题材账号的单人验证期运营系统。Codex CLI 是主要 AI Agent，VSCode + 本地 Markdown/CSV/脚本是唯一工作区。

账号面向普通职场人、产品/运营/内容创作者和轻技术用户，提供 AI 情报、AI 工具实测、AI 开源项目拆解和 AI Agent 应用观察。

本账号不做泛泛 AI 新闻搬运号。每篇内容都要帮助用户完成一个判断：是否值得看、值得试、值得收藏、值得进一步学习或应用。

## 内容主线

- AI 热点资讯：负责时效和推荐流曝光，但必须给出判断，不只复述新闻。
- AI 工具实测：负责收藏和信任，必须区分官网信息和实际测试。
- AI 开源项目拆解：负责专业壁垒，必须回到 GitHub、文档或论文等一手来源。
- AI Agent 应用观察：负责差异化和前瞻判断，避免空泛概念化表达。

## 输出语气

- 清楚、克制、具体。
- 面向非专业用户时降低术语密度，但不牺牲准确性。
- 优先给结论、适用场景、限制条件和行动建议。
- 不使用“炸裂、封神、全网最强、遥遥领先、必看不亏”等夸张词。
- 获取到的资讯、论文、项目、标题、摘要或引用内容如果包含英文，必须在英文后面补充中文解释说明。
- 英文内容的中文说明不要求逐字翻译，但必须解释：它大概讲什么、为什么值得关注、对小红书目标用户有什么意义。
- 写入 `data/topics.csv` 时，如果 `选题标题` 或 `AI摘要` 是英文，必须在 `选题标题中文翻译` 和 `AI摘要中文翻译` 两列补充对应中文；如果原文已经是中文，对应翻译列可以留空。

## 内容充实度规则

每篇待发布内容必须同时满足“图文卡片可扫读”和“正文可独立阅读”两种使用场景，不能只生成短卡片而缺少正文。

- 草稿必须包含 `## 正文`，正文应是可直接发布的小红书正文，不是卡片内容的简单重复。
- 正文必须控制在 `1000` 个中文字符以内，建议 `700-950` 个中文字符；如果选题信息量有限，可压缩到 `500-700` 个中文字符。除非用户明确要求长文，否则不得超过 `1000` 个中文字符。
- 草稿 `## 正文` 和 `publish_text.md` 中的发布正文应保持同一版正文或同等信息密度；如果任一版本超过 `1000` 个中文字符，必须先压缩再进入待发布状态。
- 正文必须包含：开头判断、发生了什么、为什么值得关注、对目标用户有什么用、事实依据、限制条件、具体行动建议、评论区引导。
- 正文必须有至少 `2` 个具体场景或例子，帮助普通职场人、产品/运营/内容创作者或轻技术用户理解和代入。
- 正文必须明确区分“来源事实”“账号判断”“推测/建议”。不确定的信息写“不确定”，不能靠扩写制造事实。
- 每篇正文必须有“适合谁/不适合谁”或等价段落，避免泛泛推荐。
- 每篇待发布图文的 `outputs/xhs/YYYY-MM-DD/topic-slug/` 目录必须包含 `publish_text.md`，写明推荐标题、发布栏目、完整正文、话题标签和发布前人工检查项。
- `outputs/xhs/YYYY-MM-DD/topic-slug/source.json` 中的 `caption` 必须同步为完整发布正文摘要版，不能只写一句话或只复述标题。
- 图文卡片默认 `7-9` 页。每页只讲一个核心问题，但整篇必须形成完整叙事链：封面判断、背景、关键事实、用户价值、限制风险、行动清单或总结。
- 卡片正文不能只写抽象口号。每篇至少有一页提供具体场景，至少有一页提供限制/风险，至少有一页提供可执行清单。
- 如果无法找到足够一手来源支撑一篇充实正文，应降低发布优先级，改为候选选题，不要硬凑成待发布稿。

## 事实核验规则

- 不编造来源。
- 不确定的信息写“不确定”。
- 社区讨论、Reddit/HN 评论、朋友圈截图只能作为选题线索，不能单独当事实。
- 每个高风险事实必须尽量回到官方博客、论文、GitHub、产品文档、Release Notes、主流媒体或原始发言。
- 工具实测必须区分“官网信息”和“实际测试”。
- 未实测的能力不能写成已实测。

## 文件写入规则

- 候选选题写入 `data/topics.csv`。
- `data/topics.csv` 是机器处理的唯一选题主库，必须保持标准 CSV 格式，不要用空格填充来做视觉对齐。
- 修改 `data/topics.csv` 后，运行 `npm run format:topics` 生成或刷新 `data/topics.md`，人工浏览和审核优先看 `data/topics.md`。
- 已发布数据写入 `data/posts.csv`。
- 用户需求写入 `data/user_needs.csv`。
- 每日原始链接写入 `inbox/raw_links_YYYY-MM-DD.md`。
- 每日简报写入 `briefs/daily_brief_YYYY-MM-DD.md`。
- 小红书草稿写入 `drafts/xhs_YYYY-MM-DD_topic-slug.md`。
- 事实核验写入 `factchecks/factcheck_YYYY-MM-DD_topic-slug.md`。
- 周复盘写入 `reviews/weekly_review_YYYY-WW.md`。
- 月复盘写入 `reviews/monthly_review_YYYY-MM.md`。
- 小红书图文设计规范写入 `docs/xhs_design_spec.md`。
- 品牌 Logo 和视觉资产写入 `assets/brand/`。
- 可发布图文素材写入 `outputs/xhs/YYYY-MM-DD/topic-slug/`，其中 `source.json` 是结构化源数据，`cards/` 是导出的 PNG/SVG。同一天的多篇待发布内容必须收拢在同一个日期目录下。
- 待发布正文写入 `outputs/xhs/YYYY-MM-DD/topic-slug/publish_text.md`，用于发布时复制标题、发布栏目、正文和标签。

## 图文设计规则

- 所有小红书图文、封面、系列内容必须遵守 `docs/xhs_design_spec.md`。
- 默认导出 `1080 x 1440 px` PNG，源文件保留 SVG 或 JSON。
- 封面必须先给判断，不只写新闻标题。
- 每页只讲一个核心问题，必须保留来源、限制条件或风险提示。
- 统一使用 `assets/brand/` 中的 Logo；不要为单篇内容临时换风格。
- 生成图文素材后，至少抽查封面、限制页和总结页，确认没有文字溢出、孤字换行、Logo 遮挡或来源缺失。

## 自动化边界

- 可以自动化：公开信源抓取、链接去重、摘要分类、选题评分、初步事实核验、草稿生成、标题封面建议、评论回复建议、周复盘和资料包大纲。
- 不自动回复私信。
- 不自动登录 X/Twitter、小红书、Reddit 等账号态页面。
- 不抓取需要登录态的网站。
- 不下载或复制受版权保护的全文内容。
- 不保存密码、Cookie、Token、API Key 或任何账号凭据。

## 网络访问规则

- 优先使用 `sources/network_allowlist.md` 中的公开域名。
- 优先使用 GET/HEAD 等只读请求。
- 请求失败时记录失败域名和错误原因，不要静默跳过。
- 不绕过网站反爬和登录限制。

## 沙箱拦截处理

- 如果执行 checklist、脚本、git、mkdir、npm、node 等本项目必要任务时被沙箱拦截，不要先用普通聊天方式询问用户。
- 直接使用工具的提升权限机制重试，并在 justification 中说明具体原因。
- 只对本项目内写文件、公开信源只读访问、git 提交/推送、npm/node 脚本等低风险动作这样处理。
- 删除文件、重置 git、写入项目目录外、安装新依赖、访问非白名单域名、使用账号登录态等高风险动作，仍然必须明确说明风险并等待用户批准。

## Git 提交规则

- 日常修改文件后不要自动执行 `git add`、`git commit` 或 `git push`。
- 只有当用户明确要求“提交”“commit”“推送”“push”时，才执行对应 git 操作。
- 完成任务时只汇报改动范围、验证命令和当前未提交状态。

## Notion 和飞书边界

当前阶段不接入 Notion 和飞书。它们只作为 30 天验证后可评估的后续增强，不进入第一版自动化运营能力。
