# 待发布正文：AI 写代码，已经不只出现在 PR 里

## 推荐标题

AI 写代码，已经不只出现在 PR 里

## 发布栏目

AI Agent 应用观察

## 正文

今天这篇 AI Agent 观察，来自一篇 arXiv 论文：`Detecting AI Coding Agents in Open Source`。

中文可以理解为：在开源中检测 AI 编程 Agent。

我的判断：这篇论文值得看，因为 AI 写代码的影响已经不只是“有人用 AI 生成了一个 PR”。在开源项目里，Agent 痕迹可能出现在普通提交、提交信息、配置文件、元数据和非 PR 工作流里。

来源事实先讲清楚。论文摘要称，它研究 AI coding agents，也就是 AI 编程 Agent，在开源开发中的采用情况，并以超过 1.8 亿个仓库作为规模背景。

最值得注意的是几个数字。摘要称，如果只用 bot-account 方法，也就是只看显性 bot 账号，只识别出 28,154 个 Claude Code 提交，占 3.3%。它还称，从 2024 年 12 月到 2026 年 4 月，每月有超过 320,000 次 Agent 相关提交；pull request census misses 79% of contributions，也就是只通过 PR 统计贡献，会漏掉很多活动。

这对普通用户有什么意义？

第一，如果你是开发者或开源维护者，未来代码审查不能只看“是不是人类账号提交”。很多 AI 辅助可能混在人类账号的普通提交里，更需要标注工具、保留日志、对高风险代码额外 review。

第二，如果你是产品经理或企业采购方，也要意识到开源依赖和内部系统的供应链风险正在变化。过去只问许可证、漏洞和维护频率，未来还要问：关键代码是否有 AI Agent 参与，审查流程是否覆盖这些情况。

两个场景很典型：公司引入开源依赖时，不能只看 star、issue 和 PR 数量，还要看维护者是否说明 AI 辅助贡献规则；团队内部使用 AI 编程 Agent 时，如果没有提交说明和高风险模块复核，容易出现“代码是 AI 写的，但没人真正理解”的问题。

限制也要保留。论文是预印本，检测方法可能存在误判或漏判。它也没有证明 AI 写的代码一定好或一定坏。更稳妥的结论是：AI 编程 Agent 已经进入软件供应链视野，审查方式要跟上。

如果你在团队里推动 AI 编程工具，建议先做 4 件事：约定提交信息是否标注 AI 辅助；高风险模块必须人工 review；保留 Agent 工具日志；把 AI 编程工具纳入安全和合规规则。

一句话总结：AI 编程 Agent 的下一步，不只是写更多代码，而是进入代码来源、审查流程和开源供应链治理。
