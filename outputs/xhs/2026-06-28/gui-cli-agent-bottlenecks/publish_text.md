## 正文

今天再补一篇 AI Agent 应用观察，来自 arXiv 论文：`GUI vs. CLI: Execution Bottlenecks in Screen-Only and Skill-Mediated Computer-Use Agents`。

中文可以理解为：比较图形界面 Agent 和命令行/技能型 Agent，看看它们分别卡在哪里。

我的判断：操作型 Agent 不能只看“会不会像人一样点屏幕”。真正落地时，更重要的是任务适合走 GUI，还是适合沉淀成技能、脚本、接口和验收器。

来源事实先讲清楚。论文比较了 `screen-only GUI agents` 和 `skill-mediated CLI agents`。前者是只看屏幕、通过点击和输入操作的 Agent；后者是通过命令行或预定义技能执行任务的 Agent。论文摘要称，评测覆盖 440 个桌面任务、18 个应用和 12 类工作流。

关键发现是：两类 Agent 的瓶颈不同。GUI Agent 的问题更偏向长程交互可靠性，比如点错位置、状态没恢复、界面变化后不知道下一步。CLI/技能型 Agent 的问题更偏向技能覆盖，如果没有合适命令、脚本或接口，它就很难完成任务。

这对普通用户有什么意义？

第一个场景：你想让 Agent 操作一个临时网页后台，界面经常变、没有 API，GUI 路线可能更自然，但要接受它可能误点、卡住、需要人工兜底。

第二个场景：你每周都要导出报表、清洗文件、改名归档、生成摘要。这类流程不一定要让 Agent 去点屏幕，更应该把步骤做成脚本、命令或固定技能，再让 Agent 调用和验收。

适合谁看：产品经理、运营、自动化工作流搭建者、想评估 Agent 落地路线的人。

不适合谁：只想看一个酷炫电脑自动操作演示的人。演示会点屏幕，不等于生产环境可靠。

限制也要保留：这是预印本论文，本账号没有复现实验；论文数字也不能直接泛化到所有软件和所有任务。

一句话总结：操作型 Agent 的关键，不是更像人，而是更稳定地完成可验收任务。

评论区可以聊：你的工作流更适合“点屏幕 Agent”，还是“技能/脚本 Agent”？
